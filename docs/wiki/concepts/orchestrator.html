Definiție

Orchestratorul primește un Problem (teorie + query + goal kind) și produce: un plan DAG de task-uri (lower/solve/check/refine/explain), apoi execută planul cu bugete și policy-uri de acceptare.

Plan DAG (model)

Lower(to UBH) ──► Solve(SAT+XOR) ──► Check(certificate/model) ──► Explain
      │
      └──► (fallback) Solve(SMT/QBF/...) ──► Check(...) ──► Explain

Bugete și UNKNOWN

  • Buget = timp/memorie/iterații; pentru metode probabilistice include seeds/confidență.
  • Dacă nu există certificat acceptabil în buget: status UNKNOWN.

Obiecte schimbate între backends

  • Lemma: constrângere derivată adăugată la teorie (cu origin).
  • UnsatCore: set de origins relevante (vezi unsat core).
  • Invariant, Strategy, CompiledKC etc.

Vezi și: Artifacts (catalog complet + verificare).

De ce e separat de session

  • Session gestionează “memoria”: fișiere + delta + cache.
  • Orchestrator gestionează “decizia”: ce tactic/backends rulează, în ce ordine.

Spec-uri relevante

  • DS-012 (planificare + caching + unknown)
  • DS-011 (policy de acceptare)
