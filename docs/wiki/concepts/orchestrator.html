Orchestrator & Query Planner

The “brain” of the system: selects backends/tactics, runs a plan DAG, and validates certificates.

DRAFT: Wiki page.
Wiki Fragment GoalKind Certificate DS-012

Definition

The orchestrator receives a Problem (theory + query + goal kind) and produces a plan DAG of tasks (lower/solve/check/refine/explain). It then executes the plan under budgets and acceptance policies.

Plan DAG (model)

Lower(to UBH) ──► Solve(SAT+XOR) ──► Check(certificate/model) ──► Explain
      │
      └──► (fallback) Solve(SMT/QBF/...) ──► Check(...) ──► Explain

Budgets and UNKNOWN

  • Budget = time/memory/iterations; for probabilistic methods it includes seeds/confidence parameters.
  • If no acceptable certificate exists within budget: status UNKNOWN.

Objects exchanged between backends

  • Lemma: a derived constraint added to the theory (with an origin).
  • UnsatCore: a set of relevant origins (see unsat core).
  • Invariant, Strategy, CompiledKC etc.

See also: Artifacts (full catalog + verification).

Why it is separate from the session

  • The session manages “memory”: files + delta + cache.
  • The orchestrator manages “decisions”: which tactics/backends to run, and in which order.
