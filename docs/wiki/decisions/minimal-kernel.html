Decision: minimal kernel + small trusted core

Separates solving (backends) from checking (trusted core) for auditability and safe evolution.

DRAFT: decision page (rationale + consequences). Normative specs remain in MD.
Wiki Vision System Spec DS-011

The decision

The kernel (trusted core) remains minimal: UBH IR + evaluation + certificate checkers. Any complex reasoning method (SAT/SMT/QBF/IC3/…) is implemented as a modular backend.

Why

  • Auditability: it is easier to trust a small checker than a complex solver.
  • Replaceability: you can swap backends without changing the system’s semantics.
  • Spec stability: input/output contracts remain the same even if the engine changes.

Engineering consequences

  • The API must expose certificates/witnesses and return UNKNOWN when they are missing.
  • You need “origin plumbing”: every constraint/artifact must retain a link to its source.
  • Backends may be heuristic/untrusted, but their results are not final without checking.

Mathematical/logical consequences

  • Solving produces claims; checking validates claims.
  • Reductions (lowering) must be sound/equisatisfiable so that certificates can be validated at the target level.
