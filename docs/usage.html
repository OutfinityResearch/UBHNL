AGISystem2 (UBH NL) Workflow

A systematic guide to defining theories, managing sessions, and generating formal proofs.

Ontological Grounding: Defining the Lexicon

The reasoning lifecycle begins with Ontological Grounding. Before rules can be authored, the domain's vocabulary must be established within a Lexicon. This involves creating a formal theory file (typically in .sys2 or .cnl) to declare the essential types, domains, and constants. By establishing this foundational dictionary, the system ensures that every subsequent statement is strictly typed and semantically consistent. This initial stage is critical for preventing lexical ambiguity and ensuring that both the human expert and the machine solver share a unified understanding of the problem space.

Interactive Experimentation: The Session Workspace

Once the foundation is established, the user initiates a Reasoning Session. This ephemeral workspace acts as an interactive laboratory. The user first loads the persistent theory files, which constitute the established "ground truth" of the domain. Within this environment, the Learn operation is used to inject transient observations or hypothetical rules. These interactive additions are local to the session, allowing for rapid "what-if" experimentation without risk to the authoritative knowledge base. Every addition is tagged with an origin ID, ensuring that any resulting reasoning can be traced back to its specific source.

Formal Inquiry: Querying the Engine

The core of the workflow is the Formal Query. The user submits a question—expressed in CNL or DSL—specifying the desired goal kind, such as proving a hypothesis (Prove), searching for a counterexample (Refute), or finding a satisfying assignment (Satisfy). The Orchestrator then analyzes the query, identifies the relevant logical fragments, and constructs an execution plan. If a conclusion is reached, the system returns a result status accompanied by a checkable witness or certificate, fulfilling the system's mandate for evidence-based reasoning.

Rational Attribution: Explaining the Result

The final stage of the lifecycle is Rational Attribution. After a query is completed, the user utilizes the Explain function to understand the derivation. By synthesizing the origin metadata of the statements involved in the proof or contradiction, the system generates a coherent explanation. This attribution allows the user to see exactly which rules from the persistent library and which observations from the interactive session contributed to the result, transforming a raw mathematical outcome into a transparent and actionable insight.

Future Tooling and Environment

While the current interface is focused on the command line and the API, the development of Integrated Authoring Environments is ongoing. These future tools will include real-time logical visualizers and guided editors that assist users in staying within the deterministic grammar of the CNL. This evolution aims to make the formal reasoning workflow even more accessible to domain experts while maintaining the absolute rigors of the underlying mathematical kernel.
